# Posture Engine — Phase 1 (Dumb Proxy)

A Vercel Edge Function that sits between Vapi and Claude. Vapi talks to it
as if it were OpenAI; it forwards each turn to Anthropic's Messages API,
translates the streaming response back into OpenAI SSE, and hands it to
the voice. No posture logic yet — Phase 1's only job is "the call sounds
identical to today, but the brain is now ours."

This is the whole risk surface (streaming + SSE translation). Nail it
before adding READ, FORCE-SET, or the Governor.

---

## Layout

```
posture-engine/
  api/chat/completions.js   the endpoint (Edge Function)
  vercel.json               rewrite so the Vapi URL can be the bare domain
  package.json              minimal, no runtime deps
  .env.example              the env vars you'll set in Vercel
```

The function lives at `/api/chat/completions`. The rewrite maps
`/chat/completions` -> that path, so Vapi's base URL can be just your
domain and Vapi will append `/chat/completions` itself.

---

## Deploy

1. Put this folder in a git repo (or run `vercel` from inside it).
2. Import the repo at vercel.com (framework preset: "Other" — there's no
   build step).
3. Add environment variables (Project -> Settings -> Environment
   Variables), from `.env.example`:
     - ANTHROPIC_API_KEY   (required)
     - ANTHROPIC_MODEL     (optional; defaults to a fast Claude)
     - MAX_TOKENS          (optional)
     - PROXY_SHARED_SECRET (leave blank for now)
4. Deploy. You'll get a URL like `https://posture-engine.vercel.app`.
5. Smoke-test the deploy in a browser — open the URL with
   `/chat/completions` on the end. A GET returns:
     `{"ok":true,"service":"posture-engine","phase":1}`
   That confirms the function is live before Vapi ever touches it.

## Wire Vapi to it

Vapi's custom LLM takes a BASE URL and POSTs to `{url}/chat/completions`,
expecting OpenAI-style streaming SSE back. Point the existing "Inbound SDR
Qualifier" assistant's model at this proxy with a PATCH.

IMPORTANT: PATCHing `model` replaces the whole model object, including the
assistant's system prompt that lives in `model.messages`. To keep the call
identical, GET the assistant first, copy its current `model.messages`, and
include them unchanged in the PATCH. (Vapi forwards those system messages
to us each turn; that's what preserves today's behavior.)

Assistant ID: `c8917a9c-dee6-4044-bf20-39212d63937d`

### Step 1 — read the current model config

```bash
curl https://api.vapi.ai/assistant/c8917a9c-dee6-4044-bf20-39212d63937d \
  -H "Authorization: Bearer $VAPI_PRIVATE_KEY" | jq '.model'
```

Copy the `messages` array from the output (your system prompt).

### Step 2 — switch the provider to the proxy

```bash
curl -X PATCH \
  https://api.vapi.ai/assistant/c8917a9c-dee6-4044-bf20-39212d63937d \
  -H "Authorization: Bearer $VAPI_PRIVATE_KEY" \
  -H "Content-Type: application/json" \
  -d '{
    "model": {
      "provider": "custom-llm",
      "url": "https://posture-engine.vercel.app",
      "model": "claude-via-posture-engine",
      "messages": [
        { "role": "system", "content": "PASTE YOUR EXISTING PROMPT HERE" }
      ]
    }
  }'
```

Notes:
  - `url` is the BARE domain. Vapi appends `/chat/completions`.
  - `model` here is just a label Vapi echoes back in the request body; the
    proxy ignores it and uses ANTHROPIC_MODEL. Keep it for clarity.
  - Replace the `messages` block with the array you copied in Step 1.

### Reverting

Keep the original `model` object from Step 1 saved somewhere. To roll
back, PATCH it straight back in. Nothing here is destructive as long as
you have that copy.

## Test

1. Place a call to the assistant (or use the Vapi dashboard's test call /
   your Calendly demo link).
2. The call should behave as it did before. Behind the scenes every line
   is now generated by Claude through this proxy.
3. Watch Vercel's function logs (Project -> Logs) for incoming POSTs and
   any 502s (upstream Anthropic errors) or 401s (auth).

If you hear a noticeable pause before lines, that's first-token latency —
try a faster ANTHROPIC_MODEL or lower MAX_TOKENS. The stream itself starts
emitting as soon as Claude produces its first token, so there's no extra
buffering on our side.

## What changes in later phases (not built yet)

  - Phase 2 — READ: inject a hardcoded posture directive into the system
    prompt and confirm the Host's stance audibly shifts.
  - Phase 3 — FORCE-SET + STORE: add the `call_posture` Supabase row and a
    keyword/regex rule table over the latest caller utterance. Posture
    moves mid-call, instant and rule-driven. Shippable on its own.
  - Phase 4 — GOVERNOR: a background Haiku-class classifier, run via
    `waitUntil()` so the voice never waits on it. Last write wins.
  - Phase 5 — TELEMETRY: append transitions to `posture_log`, surface
    current posture per active character in Director's View.

The endpoint is structured so all of that bolts on around the same READ /
FORCE-SET / stream-the-Host-line spine that's already here.
